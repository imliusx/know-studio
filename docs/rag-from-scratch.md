# 从零理解 know-studio：RAG 与本项目完整讲解

> 写给完全不懂 RAG 的人。读完你能：① 用大白话讲清 RAG 是什么、解决什么问题；② 看懂这个项目每一步在干嘛；③ 理解 RRF、聚簇、证据分级这些"黑话"。
>
> 全程用类比 + 项目真实代码，不堆术语。

---

## 第 0 步：先建立一个直觉——为什么需要 RAG？

### 一个生活类比：开卷考试

想象你在参加一场考试：

- **闭卷考试**：所有知识都得背在脑子里。题太难或没背过，你就只能**瞎编**。
- **开卷考试**：可以翻书。翻到相关那页，照着抄/总结，答得又准又稳。

**大模型（ChatGPT 这种）默认是"闭卷考试"。** 它训练时读过海量文本，知识"记"在参数里，但：
1. **它不知道你公司的内部资料**（你的产品文档、规章制度、合同……它从没见过）。
2. **它会瞎编**（行业黑话叫"幻觉"）：没把握也一本正经地胡说。
3. **它的知识有截止日期**，最新的不知道。

**RAG（Retrieval-Augmented Generation，检索增强生成）就是把大模型从"闭卷"变成"开卷"。**

> 一句话：**RAG = 先去你的资料库里翻相关内容，再把翻到的内容 + 用户问题一起塞给大模型，让它"照着资料回答"。**

### RAG 这四个字母怎么拆

- **R**etrieval = **检索**：去资料库翻相关内容（核心难点）
- **A**ugmented = **增强**：把翻到的内容当作"参考资料"附上去
- **G**eneration = **生成**：大模型基于参考资料生成回答

所以 RAG 的重点其实不在"大模型"，而在**前面那个"检索"怎么做**——这正是 know-studio 项目花大力气的地方。

## 第 1 步：那"资料库"长什么样？——向量化

新的问题：你的资料是一堆 Word/PDF/网页，**电脑怎么知道"哪段和用户问题相关"？**

### 1.1 人怎么判断"相关"？
你问"年假怎么休"，你会自动找到写着"员工年假规定"那段，因为**语义相近**。

### 1.2 电脑怎么判断"语义相近"？——把文字变成数字
有一种技术叫 **Embedding（嵌入/向量化）**：把任意一段文字，通过一个模型，变成**一串数字（向量）**。关键性质：

> **语义相近的文字，变出来的数字也相近。**

类比：给每段文字算一个"GPS 坐标"。语义相近的内容，坐标就靠得近。
- "年假规定" 的坐标可能是 `(0.8, 0.2, ...)`
- "带薪休假政策" 的坐标可能是 `(0.79, 0.21, ...)` ← 靠得很近！虽然字不一样
- "食堂菜单" 的坐标可能是 `(0.1, 0.9, ...)` ← 离得远

于是"找相关内容"就变成了"**在坐标空间里找最近的邻居**"——这叫**向量检索**（语义检索）。

### 1.3 项目里这一步用什么？
- 模型：**Ollama**（本地跑的 embedding 模型，把文字变成向量）
- 存储：**pgvector**（PostgreSQL 的向量插件，专门存这些"坐标"并能快速找邻居）

## 第 2 步：资料太长怎么办？——切片（Chunking）

你的文档可能 100 页，不可能整篇塞给模型（太长、超 token、还贵）。所以要先**切片**。

### 类比：把一本厚书拆成一张张卡片
把整篇文档切成几百个小片段（chunk），每个片段大概几百字。检索时只捞最相关的几张卡片。

### 项目真实配置（`ChunkingProperties`）
```
targetTokens = 500   目标：每片约 500 token（大致几百字）
maxTokens    = 800   上限：最多 800 token，超了强制切
overlapTokens = 80   重叠：相邻两片有 80 token 重叠
```

**overlap（重叠）是个关键细节**，用图说明：

```
不重叠（坏）：片A 结尾 | 片B 开头
              ↑ 一句话正好被从中间切断，两边都不完整

有重叠（好）：片A [======重叠80======]
             片B    [======重叠80======]
              ↑ 切断的地方，重叠区能保证至少有一片是完整的
```

项目用的是**结构感知切片**（`StructureAwareChunkTransformer`）——会尽量沿着标题、段落边界切，而不是死板地数到 500 字就一刀。

## 第 3 步：检索的难题——光靠"语义相近"不够

到这里你已经有了一套基础 RAG：文档切片 → 向量化 → 用户提问也向量化 → 找最近的几张卡片 → 塞给模型。**但 know-studio 没停在这里，因为单纯向量检索有硬伤。**

### 3.1 向量检索的硬伤：精确词抓不住
问："**DDR4** 内存能换 **DDR5** 吗？"
- 向量模型觉得 "DDR4/DDR5/内存兼容性" 都差不多 → 可能召回一堆"内存"相关内容，但漏掉精确讲 DDR4 vs DDR5 的那段。

### 3.2 那用传统关键词搜索（比如 Elasticsearch）行吗？
关键词搜索（学术名 **BM25**）：就是"全文搜单词"，像 Ctrl+F 但更智能（按词频打分）。
- 它能精准抓住 "DDR4""DDR5" 这种词。
- 但它不懂同义词：问"内存条兼容性"，文档里写"DRAM 互换"，它就找不到了。

### 3.3 项目的选择：两个都要（混合检索 / Hybrid）
> **语义检索（pgvector）抓"意思相近"，关键词检索（Elasticsearch BM25）抓"字面精确"，两路同时召回，各有所长。**

| 检索方式 | 实现 | 擅长 | 弱项 |
|---|---|---|---|
| 语义检索 | pgvector + Ollama | 同义改写、概念相近 | 精确 token、专有名词 |
| 关键词检索 | Elasticsearch + IK 分词 | 精确词、型号、缩写 | 同义词、换种说法 |

## 第 4 步：两路结果怎么合并？——RRF 登场（重点！）

现在你从两路各捞回来一堆结果，**问题来了：它们怎么排到一起？**

### 4.1 为什么不能直接比分数？
- 向量检索给的相似度分：**0~1 之间**（比如 0.87）
- 关键词 BM25 给的分：**没有上限**（可能 3.5，也可能 15.8）

这就像：
- 一个老师用 10 分制打分，一个老师用 100 分制打分。
- 直接比"谁分高"毫无意义——15.8（BM25）和 0.87（向量）根本不在一个尺度上。

如果硬要合并，你得先"归一化"再"调权重"（向量占几成、关键词占几成），这个权重**极其难调**，玄学。

### 4.2 RRF 的妙招：只看排名，不看分数
**RRF = Reciprocal Rank Fusion（倒数排名融合）**。它的核心思想极其朴素：

> **不看分数，只看每个结果在各自那路里排第几名。排名越靠前，加分越多。最后把两路的加分加起来。**

公式（你不用记，理解意思就行）：
```
某 chunk 的最终分 =  1/(60 + 在向量路的名次)  +  1/(60 + 在关键词路的名次)
```

#### 用一个具体例子讲透
假设"DDR4换DDR5"那一段（叫它 chunk X）：

| | 向量路排名 | 关键词路排名 |
|---|---|---|
| chunk X | 第 1 名 | 第 2 名 |
| chunk Y | 第 2 名 | （没命中） |

算分（K=60）：
```
chunk X 分 = 1/(60+1) + 1/(60+2) = 1/61 + 1/62 ≈ 0.0164 + 0.0161 = 0.0325
chunk Y 分 = 1/(60+2) + 0       = 1/62        ≈ 0.0161
```
→ **chunk X 胜出**，因为它在两路里都靠前。

#### 为什么这个公式长这样？
- `1/(60+rank)`：名次越靠后，分越小，但**衰减得平缓**（因为有 +60 垫底）。
- 为什么是 60？业界经验值（Elasticsearch 8 的混合检索也用类似值）。你可以理解成"给尾部名次一点体面的分，不让它们归零"。
- **两路都命中的 chunk**（像 X）天然占便宜，因为两个加法项都有值——这正好是我们想要的：**两种检索方式都觉得它相关，那它大概率就是真相关。**

#### 一句话总结 RRF
> RRF 就是"**别比分数了，比排名**"，完美绕开了"两路分数尺度不一样"的死结，而且**不用调任何权重**。

### 4.3 项目里更进一步：一个变多个（查询规划）
你问一个问题，项目不会只拿原问题去检索，而是先让大模型帮你**改写出几个版本**（`QueryPlanningService`）：

```
DIRECT（直查）   ：直接用原问题
REWRITE（改写）  ：换个说法（"年假怎么休"→"员工带薪休假申请流程"）
DECOMPOSE（拆分）：拆成几个子问题
```
**每个版本都跑一遍双通道检索，结果全合并。** 等于撒更大的网，召回率更高。

## 第 5 步：召回的片段是"碎的"——聚簇 + 扩窗

### 5.1 问题：切片切碎了语义
还记得切片吗？一个完整的论点可能被切成了 chunk 3、4、5 三片。如果检索命中的正好是第 4 片，模型只看到第 4 片，**前因后果是断的**。

### 5.2 解决：聚簇（Cluster）
项目把**同一个文档里、序号相邻**的命中合并成一组（叫一个"簇"）。
- 命中了文档 A 的第 3、4、5 片 → 合并成**一条证据**（而不是三条重复的）。
- 用簇里得分最高的那片当代表（primary）。

### 5.3 再解决：邻居扩窗（Window）
簇确定后，**前后再各多读 1 片**（`neighborWindow=1`），拼成更完整的上下文：
```
命中簇：       [第4片]
扩窗后喂模型：[第3片][第4片][第5片]   ← 上下文补全了
```
类比：你在书里翻到一个关键句，会习惯性地**前后各读一段**确认语境——这正是扩窗在做的事。

> 这套思想在学术界叫 **Parent Document Retrieval（父文档检索）**。项目没用重型方案，而是用"序号相邻就拼窗"做了个轻量版。

## 第 6 步：防止大模型瞎编——证据分级 + 拒答（防幻觉核心）

### 6.1 最头疼的问题：幻觉
就算你把检索到的资料塞给大模型，如果资料**和问题其实不太相关**，模型还是会硬着头皮编。企业场景里**答错比不答严重 10 倍**（给客户错误的合规建议 = 出事）。

### 6.2 项目的硬约束：先给证据打分级
检索完，先算证据"够不够硬"，分**四级**（`EvidenceLevel`）：

| 级别 | 判定条件（项目真实逻辑） | 含义 |
|---|---|---|
| **SUFFICIENT** | 证据≥2条 且（两路都命中 OR 向量分≥0.95） | 证据充分 |
| **PARTIAL** | 有多条但不够强 | 只覆盖一部分 |
| **WEAK** | 只有一条弱证据 | 相关性有限 |
| **NONE** | 啥都没召回到 | 没证据 |

### 6.3 分级决定"怎么回答"
每一级对应一段**给大模型的指令**（拼在 prompt 里）：

- SUFFICIENT → "可以正常回答，但不得超出证据臆测"
- PARTIAL → **"只答证据明确支持的部分，没覆盖的必须说不知道"**
- WEAK → "必须说明依据有限，别下确定结论"
- NONE → **"没有证据，必须直接拒答"**

### 6.4 为什么这是真功夫？
很多人防幻觉只是在 prompt 里写一句"请不要编造"——**这根本拦不住**。项目是在**送进模型之前**用规则硬判证据够不够，**不够就不让它答**。这才是"可信"和"玩具"的分水岭。

---

## 第 7 步：回答要能追溯——引用（Citation）

每次回答都附带 `citations`（引用清单）：文件名、片段、相关度分数。用户能看到"这个回答是从哪几个文件来的"。**这是企业知识库的基本要求——答案必须可追溯、可审计。**

## 第 8 步：把上面所有步骤串起来（一张图）

```
① 用户上传文档
   → 切片（500 token/片，有重叠）
   → 向量化（Ollama）存 pgvector
   → 同时建关键词索引（Elasticsearch）
   → 状态置为 READY

② 用户提问："DDR4 能换 DDR5 吗"
   │
   ├─ 查询规划：改写出多个 query（DIRECT/REWRITE/DECOMPOSE）
   │
   ├─ 每个 query 双通道召回（各 topK=50）：
   │     ├─ 向量路（pgvector）—— 抓语义
   │     └─ 关键词路（ES BM25）—— 抓精确词
   │
   ├─ RRF 融合：按排名合并两路，排个总名次
   │
   ├─ 聚簇 + 扩窗：相邻碎片合并，前后补 1 片
   │
   ├─ 证据分级：SUFFICIENT / PARTIAL / WEAK / NONE
   │
   ├─ 拼 prompt = [证据片段] + [分级指令] + [用户问题]
   │
   └─ 大模型生成回答（带 citations）
       够证据 → 答；不够 → 拒答
```

## 第 9 步：Assistant（AI 助手）和 QA 有什么区别？

项目里有两个"问答"入口，别搞混：

| | QA（知识库问答） | Assistant（AI 助手） |
|---|---|---|
| 形态 | 一问一答 | 多轮对话（有记忆） |
| 检索 | 固定每次都检索 | **把检索做成"工具"，模型自己决定要不要用** |
| 特点 | 严约束，没证据就拒答 | 可以闲聊，需要查资料时才调工具 |

### "把检索做成工具"是什么意思？（Tool Calling）
在 Assistant 里，知识库检索被包装成一个叫 `knowledgeBaseSearch` 的**工具**，大模型可以**自己决定**："这个问题我得查资料"→ 主动调用这个工具 → 拿到结果 → 再回答。

类比：QA 像你强制让员工"每个问题都必须先翻档案"；Assistant 像一个聪明员工，**平时聊天不用翻档案，遇到事实问题才去翻**。

项目还有两个细节设计：
- **防重复调用**：如果这一轮已经查过一次，就不让模型再查（`DUPLICATE_TOOL_CALL`），防止它陷入死循环反复查、烧钱。
- **短期记忆重建**：多轮对话历史太长会超 token，项目用"摘要 + 最近 10 条"重建上下文，而不是无脑全塞。

## 名词速查表（黑话翻译）

| 黑话 | 大白话 |
|---|---|
| **RAG** | 开卷考试式问答：先翻资料再回答 |
| **Embedding / 向量化** | 把文字变成一串数字（坐标），语义相近则坐标相近 |
| **pgvector** | 数据库里专门存/查这些坐标的插件 |
| **Chunk / 切片** | 把长文档切成小片段 |
| **Overlap / 重叠** | 相邻片段重叠一部分，避免切断句子 |
| **BM25** | 经典关键词搜索打分算法（Ctrl+F 的智能版） |
| **混合检索 Hybrid** | 语义 + 关键词两路一起搜 |
| **RRF** | 只比排名不比分数，把两路结果合并的算法 |
| **topK** | 每路只取前 K 个结果（项目 K=50） |
| **聚簇 Cluster** | 把相邻的碎片合并成一条 |
| **扩窗 Window** | 前后多读几片补全上下文 |
| **证据分级** | 按证据强度分四级，决定答还是拒答 |
| **幻觉** | 大模型没证据也瞎编 |
| **Citation / 引用** | 答案附带的出处清单 |
| **Tool Calling** | 让大模型自己决定要不要调用某个功能（如检索） |
| **token** | 大模型处理文字的最小单位（约 0.75 个英文单词，中文一字约 1~2 token） |

## 最后：这个项目牛在哪？（一句话版）

> 大多数 RAG demo 就是 **"切片→向量检索 topK→塞给模型"**，三步完事。know-studio 牛在它把检索这一层做厚了：**多查询规划 → 双通道召回 → RRF 融合 → 聚簇扩窗 → 证据分级拒答**，每一步都在解决一个真实的工程问题（召回不全、分数不可比、上下文断裂、模型瞎编）。**这些"把简单 demo 推到工程可用"的细节，正是项目的技术含量所在。**
